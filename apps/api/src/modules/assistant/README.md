# OA智能助手模块

## 概述

基于PRD文档设计的智能代理系统，实现自然语言交互的OA申请流程处理。

## 核心特性

### 1. 上下文管理

#### 会话上下文 (Session Context)
- **作用域**: 单次对话会话
- **生命周期**: 会话开始到结束
- **内容**: 对话历史、临时变量、会话状态
- **隔离性**: 完全隔离，不同会话互不影响

#### 流程上下文 (Process Context)
- **作用域**: 单个申请流程
- **生命周期**: 流程启动到完成/取消
- **内容**: 流程类型、已收集参数、流程状态、流程ID
- **隔离性**: 不同流程类型完全隔离

#### 共享上下文 (Shared Context)
- **作用域**: 全局
- **生命周期**: 持久化存储
- **内容**: 用户基础信息、常用配置、历史申请记录、用户偏好
- **共享性**: 所有流程可读取，自动预填充

### 2. 流程状态机

```
INITIALIZED (初始化)
    ↓
PARAMETER_COLLECTION (参数收集中)
    ↓
PENDING_CONFIRMATION (待确认)
    ↓
EXECUTING (执行中)
    ↓
COMPLETED (已完成) / FAILED (失败) / CANCELLED (已取消)
```

### 3. 智能体架构

#### IntentAgent (意图识别)
- 支持LLM和规则两种模式
- 识别8种意图类型
- 提取实体信息（金额、日期、流程类型等）

#### FlowAgent (流程匹配)
- 基于关键词匹配流程
- 智能消歧义
- 置信度评估

#### FormAgent (表单提取)
- 从自然语言提取字段值
- 支持多种字段类型
- 生成智能提问

## API接口

### POST /v1/assistant/chat

对话工作台核心接口，支持自然语言交互。

**请求参数:**
```json
{
  "message": "我要报销差旅费",
  "sessionId": "optional-session-id",
  "userId": "optional-user-id",
  "tenantId": "optional-tenant-id"
}
```

**响应示例:**
```json
{
  "sessionId": "session-123",
  "message": "正在为您填写\"差旅费报销\"。\n\n请问报销金额是多少？",
  "intent": "create_submission",
  "needsInput": true,
  "formData": {
    "employeeId": "auto-filled"
  },
  "missingFields": [
    {
      "key": "amount",
      "label": "报销金额",
      "question": "请问报销金额是多少？"
    }
  ],
  "processStatus": "parameter_collection"
}
```

### GET /v1/assistant/sessions

获取用户的会话列表。

**查询参数:**
- `tenantId`: 租户ID（可选）
- `userId`: 用户ID（可选）

### GET /v1/assistant/sessions/:sessionId/messages

获取指定会话的消息历史。

### DELETE /v1/assistant/sessions/:sessionId

删除指定会话及其所有消息。

### POST /v1/assistant/sessions/:sessionId/reset

重置会话上下文，清除流程状态但保留消息历史。

## 支持的意图类型

1. **CREATE_SUBMISSION** - 创建申请
   - 示例: "我要报销"、"申请请假"、"发起采购"

2. **QUERY_STATUS** - 查询状态
   - 示例: "我的申请到哪了"、"查询进度"

3. **CANCEL_SUBMISSION** - 撤回申请
   - 示例: "撤回申请"、"取消"

4. **URGE** - 催办
   - 示例: "催办"、"催一下"

5. **SUPPLEMENT** - 补件
   - 示例: "补件"、"补充材料"

6. **DELEGATE** - 转办
   - 示例: "转办"、"委托"

7. **SERVICE_REQUEST** - 服务请求
   - 示例: "有什么流程"、"可以办什么"

8. **UNKNOWN** - 未知意图

## 参数收集流程

### 智能预填充
系统自动从共享上下文读取以下字段：
- `employeeId` / `applicantId` - 员工ID
- `applicantName` / `name` - 申请人姓名
- `department` - 部门
- `approver` / `approverId` - 审批人
- `cc` / `ccList` - 抄送人

### 交互式收集
对于缺失的必填字段，系统会：
1. 生成针对性问题
2. 验证用户输入
3. 提供友好的错误提示
4. 支持修改已填写的字段

### 确认机制
参数收集完成后：
1. 生成草稿
2. 展示表单摘要
3. 等待用户确认
4. 支持修改或取消

## 错误处理与回滚

### 错误类型
- **参数验证错误**: 提示用户重新输入
- **MCP调用失败**: 自动重试（最多3次）
- **网络超时**: 提示用户稍后重试
- **权限错误**: 提示用户联系管理员
- **业务规则错误**: 解释原因并引导修正

### 回滚策略
- 流程执行失败时，清理已创建的临时数据
- 保留流程上下文，允许用户修改后重试
- 记录失败日志，便于问题排查

## 审计日志

所有关键操作都会记录审计日志：
- 意图识别
- 流程初始化
- 参数收集
- 申请提交
- 操作执行（撤回、催办等）
- 错误和异常

## 使用示例

### 完整的请假申请流程

```
用户: "我要请假"
助手: "正在为您填写\"请假申请\"。\n\n请问您要请什么类型的假？（年假/病假/事假/其他）"

用户: "年假"
助手: "请问请假的起止时间是？"

用户: "3月10日到3月12日"
助手: "请简要说明请假原因"

用户: "家里有事需要处理"
助手: "\"请假申请\"草稿已生成。\n\n表单内容：\n  类型: 年假\n  时间: 2026-03-10 至 2026-03-12（共3天）\n  原因: 家里有事需要处理\n  审批人: 张经理\n\n确认提交吗？"

用户: "确认"
助手: "申请已提交成功！\n\n申请编号：REQ20260305001\n流程：请假申请\n\n您可以随时查询申请进度。"
```

### 查询申请状态

```
用户: "我的申请到哪了"
助手: "您最近的申请：\n1. 请假申请 - 已提交 (2026-03-05)\n2. 差旅费报销 - 已提交 (2026-03-01)\n\n需要查看详情吗？"
```

### 催办申请

```
用户: "催办 REQ20260305001"
助手: "催办操作已成功执行！\n\n申请：请假申请\n编号：REQ20260305001"
```

## 配置选项

### 环境变量

- `USE_LLM_FOR_INTENT`: 是否使用LLM进行意图识别（默认: true）
- `DEFAULT_TENANT_ID`: 默认租户ID
- `LLM_API_KEY`: LLM API密钥
- `LLM_MODEL`: LLM模型名称

## 扩展性

### 新增申请类型

1. 在ProcessLibrary中定义流程模板
2. 配置字段Schema
3. 实现MCP接口
4. 系统自动支持新流程

### 自定义流程

支持通过配置文件定义新流程：
- 流程定义
- 参数定义
- 验证规则
- 权限规则
