# 15 企业级OA接入与同步架构设计

**文档版本**: v1.0  
**创建日期**: 2026-03-09  
**适用范围**: `OA_agent` 企业级接入、同步、提交流程治理

---

## 15.1 目标

把当前系统从“API 文档解析 + 流程发布”升级为“企业级 OA 接入与运行平台”。

### 核心目标

1. 支持多租户、多 OA 系统并行接入。
2. 支持流程元数据、主数据、运行状态三类同步。
3. 支持流程版本化、灰度发布、漂移检测、人工审核。
4. 支持高可靠提交、事件回放、幂等处理、Webhook 接入。
5. 支持企业级安全治理，包括密钥托管、审计、隔离、告警。

### 设计原则

1. `OA 是事实源`，平台不做主数据主库。
2. `提交链路短，重同步异步化`，所有重操作走队列。
3. `事件优先`，状态变化先落事件，再更新快照。
4. `版本优先于覆盖`，任何外部流程变化默认生成新版本而不是直接覆盖。
5. `能力驱动`，不同 OA 按 capability 降级，不强行统一成同一种接入方式。

## 15.2 目标架构

### 四层架构

#### 1. 控制面 Control Plane

负责接入和治理：

- Connector 管理
- 凭证管理
- 能力声明
- 流程目录与模板版本管理
- 同步策略配置
- 审核发布

#### 2. 数据面 Data Plane

负责生产运行：

- 提交申请
- 查询状态
- 撤回、催办、补件、转办
- 引用数据读取
- OA API 调用编排

#### 3. 同步面 Sync Plane

负责最终一致性：

- Schema Sync
- Master Data Sync
- Status Sync
- Webhook Inbox
- Drift Detection

#### 4. 治理面 Governance Plane

负责企业能力：

- 审计日志
- 指标与告警
- 链路追踪
- SLA/SLO
- 安全策略
- 数据保留与脱敏

## 15.3 企业级领域模型

### 关键对象

#### Connector

表示一个租户接入的一个 OA 实例。

职责：

- 定义基础地址、厂商、环境、状态
- 绑定凭证引用
- 绑定同步策略
- 绑定能力模型

#### RemoteProcess

表示 OA 原生流程对象，是外部流程的稳定锚点。

职责：

- 记录 `connectorId + remoteProcessId`
- 保存外部流程原始元数据
- 保存 source hash
- 关联平台内部版本化模板

#### ProcessTemplate

表示平台内部可编排、可对话、可发布的标准流程模板版本。

职责：

- 固化一个版本的字段、规则、权限、UI hints
- 支持发布、废弃、回滚
- 支持灰度和人工审核

#### ReferenceDataset / ReferenceItem

表示部门、人员、假别、科目、审批节点、选项字典等引用数据。

职责：

- 提供字段选项、映射和缓存
- 支持增量同步或 TTL 刷新

#### Submission / SubmissionEvent

表示运行态单据与事件流。

职责：

- `Submission` 存当前快照
- `SubmissionEvent` 存完整事件历史
- 支持幂等、补偿、回放

#### SyncCursor / SyncJob

表示每个连接器每个同步域的进度和运行记录。

职责：

- 保存 cursor / etag / token / lastWatermark
- 支持增量同步恢复
- 支持失败重试和观察性

#### DriftEvent

表示外部 OA 变化。

职责：

- 字段变更
- 接口变更
- 权限变更
- 字典变更
- 认证变更

## 15.4 同步分层设计

### 15.4.1 流程元数据同步 Schema Sync

同步对象：

- 流程目录
- 表单字段
- 校验规则
- 节点与路由
- 操作能力

同步策略：

1. `discover processes`
2. `pull schema`
3. `normalize`
4. `sourceHash`
5. `diff`
6. `auto publish / review`

处理规则：

- 只新增字段：自动升级小版本
- 修改字段类型、删除字段、权限收紧：进入人工审核
- 外部流程下线：标记 `deprecated`，不直接删除

### 15.4.2 主数据同步 Master Data Sync

同步对象：

- 用户
- 部门
- 岗位
- 组织树
- 流程字典项
- 审批人候选集

同步策略：

- 优先使用 OA 增量接口
- 无增量接口则使用快照对比
- 高频字典项走缓存 + TTL 刷新
- 用户和权限类数据保留最小必要字段

### 15.4.3 运行状态同步 Status Sync

同步对象：

- 单据受理状态
- 当前审批节点
- 审批意见
- 轨迹日志
- 撤回/补件/转办结果

同步策略：

- 支持 webhook 的 OA：优先 webhook
- 不支持 webhook：只轮询 `pending/submitted/in_progress/supplementing`
- 所有回调先进入 `WebhookInbox/EventInbox`
- 事件去重后写入 `SubmissionEvent`
- 再异步更新 `Submission.currentStatus`

## 15.5 Prisma 表设计建议

以下为基于当前 schema 的增量设计，目标是兼容现有 `Connector`、`ProcessTemplate`、`Submission`。

### 15.5.1 Connector 扩展


### 15.5.2 外部流程锚点

### 15.5.3 流程模板扩展

建议在现有 `ProcessTemplate` 上增加以下字段：

```prisma
model ProcessTemplate {
  remoteProcessId   String?
  sourceHash        String?
  sourceVersion     String?
  reviewStatus      String   @default("approved") // draft, review, approved, rejected
  changeSummary     Json?
  supersedesId      String?
  lastSyncedAt      DateTime?
  publishedBy       String?

  @@index([connectorId, processCode, status])
  @@index([connectorId, remoteProcessId, version])
}
```

唯一约束建议从：

```text
(tenantId, processCode, version)
```

调整为：

```text
(connectorId, processCode, version)
或
(connectorId, remoteProcessId, version)
```

原因：

- 不同 OA 供应商可能存在相同 `processCode`
- 企业级并行接入时，流程唯一性必须落在连接器维度

## 15.7 适配器接口升级建议

当前适配器接口需要从“仅支持 discover/submit/queryStatus”升级为企业级运行接口。

### 建议接口

```typescript
interface EnterpriseOAAdapter {
  getCapabilities(): Promise<ConnectorCapabilities>;
  healthCheck(): Promise<HealthCheckResult>;

  listProcesses(input?: { cursor?: string }): Promise<RemoteProcessPage>;
  getProcessSchema(remoteProcessId: string): Promise<RemoteProcessSchema>;

  listReferenceDatasets(): Promise<ReferenceDatasetMeta[]>;
  pullReferenceItems(datasetCode: string, cursor?: string): Promise<ReferencePage>;

  submit(request: SubmitRequest): Promise<SubmitResult>;
  queryStatus(submissionId: string): Promise<StatusResult>;
  listSubmissionUpdates(input: { cursor?: string; statuses?: string[] }): Promise<SubmissionUpdatePage>;

  cancel?(submissionId: string): Promise<CancelResult>;
  urge?(submissionId: string): Promise<UrgeResult>;
  delegate?(request: DelegateRequest): Promise<DelegateResult>;
  supplement?(request: SupplementRequest): Promise<SupplementResult>;

  verifyWebhook?(headers: Record<string, string>, body: unknown): Promise<WebhookVerifyResult>;
}
```

### 适配器能力分级

| 能力等级 | 说明 | 适用场景 |
|----------|------|----------|
| L1 | 仅可提交和查状态 | 老旧 OA |
| L2 | 可读流程 schema | 常规 OpenAPI OA |
| L3 | 可拉引用数据 | 可做字段选项联动 |
| L4 | 支持增量同步 | 企业级稳定接入 |
| L5 | 支持 Webhook + 实时权限 | 深度集成 OA |

## 15.8 关键运行流程

### 15.8.1 新 OA 接入

```text
创建 Connector
-> 录入 SecretRef
-> 识别 Capability
-> 执行 Bootstrap
-> 建立 RemoteProcess
-> 生成首版 ProcessTemplate
-> 管理员审核发布
```

### 15.8.2 日常 Schema Sync

```text
Scheduler 触发
-> 拉取流程目录
-> 拉取 schema
-> 计算 sourceHash
-> Diff
-> 自动生成新版本或进入 review
-> 记录 DriftEvent
-> 更新 SyncCursor
```

### 15.8.3 提交与状态回写

```text
用户确认提交
-> 幂等校验
-> 创建 Submission
-> 入 submit queue
-> Adapter submit
-> 写 SubmissionEvent(submitted)
-> 更新 Submission 快照
-> 入 status sync queue
-> pull/webhook 回写
```

### 15.8.4 Webhook 处理

```text
OA 回调
-> 验签
-> 落 WebhookInbox
-> 去重
-> 映射到 submission
-> 写 SubmissionEvent
-> 更新 Submission.currentStatus
-> 触发消息通知/审计
```

## 15.9 安全与合规要求

### 必做项

1. 数据库不存明文密码、token、cookie。
2. 所有外部凭证统一走 Vault/KMS/Secrets Manager。
3. Webhook 必须支持签名校验、时间戳校验、防重放。
4. 审计日志不可被业务接口直接修改。
5. 所有租户数据必须带 `tenantId` 强约束。
6. 用户可见日志和调试日志必须脱敏。
7. 连接器调用应支持白名单出口、超时、重试、熔断。

### 建议项

- 敏感字段列级加密
- 操作审计分级
- 密钥自动轮换
- 提交结果签名快照
- 合规保留策略

## 15.12 对当前项目的直接改造建议

### 优先修改点

1. `ProcessTemplate` 唯一性改到 `connectorId` 维度。
2. 发布逻辑改成版本递增，禁止固定 `version = 1`。
3. `SubmissionStatus` 升级为 `SubmissionEvent + current snapshot` 双模型。
4. 新增 `sync` 队列，不再在请求链路里做重同步。
5. `Connector.authConfig` 仅保留非敏感配置，凭证迁移到 `ConnectorSecretRef`。
6. 适配器工厂按 connector capability 实例化真实 adapter，而不是 mock。

### 不建议继续的做法

1. 不要把所有 OA 用户、组织、流程明细全量长期复制到本地。
2. 不要把 LLM 放在提交主链路中做最终判定。
3. 不要在同一张表里同时承载“外部流程对象”和“内部版本模板”。
4. 不要以 `tenantId + processCode` 作为跨 OA 唯一锚点。

## 15.13 推荐下一步

建议按以下顺序继续落地：

1. 先改 Prisma：补 `ConnectorSecretRef`、`ConnectorCapability`、`RemoteProcess`、`ReferenceDataset`、`ReferenceItem`、`SyncCursor`、`SyncJob`、`SubmissionEvent`、`WebhookInbox`。
2. 再改 adapter 接口与工厂。
3. 再落 `sync` 模块和三个 worker。
4. 最后改提交与状态查询主链路。

这四步完成后，系统才算具备企业级接入平台的基础形态。
