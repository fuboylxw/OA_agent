## 8.1 权限模型

### 双层校验架构

```
用户请求
  │
  ▼
┌─────────────────────┐
│ 第一层：平台预校验   │  ← RBAC + ABAC（本地，快速）
│ 角色白名单 + 属性规则 │
└─────────┬───────────┘
          │ allowed
          ▼
┌─────────────────────┐
│ 第二层：OA实时校验   │  ← 调用OA接口（远程，最终判定）
│ 提交前最终确认       │
└─────────┬───────────┘
          │ allowed
          ▼
       执行操作
```

**设计原则**：
- 平台预校验快速过滤80%+无权限请求，减少OA调用
- OA实时校验作为最终判定，保证权限准确性
- 任一层拒绝即拒绝，两层都通过才允许

## 8.4 属性权限（ABAC）- P2

### 属性维度

| 属性 | 来源 | 示例 |
|------|------|------|
| user.dept | 用户部门 | 计算机学院 |
| user.title | 用户职称 | 副教授 |
| user.budget_limit | 预算额度 | 50000 |
| user.employment_type | 用工类型 | 正式/临时 |
| user.campus | 所在校区 | 主校区 |
| form.amount | 表单金额 | 1200 |
| form.dept | 表单部门 | 计算机学院 |

### 规则表达式示例

```
# 只能报销本部门费用
user.dept == form.dept

# 金额不超过个人预算额度
form.amount <= user.budget_limit

# 正式员工才能申请出差
user.employment_type == "正式"

# 组合规则
user.dept == form.dept && form.amount <= user.budget_limit
```

## 8.5 权限校验流程

### 校验流程

```
1. 接收权限校验请求
   ├── userId
   ├── processCode
   └── action (submit/cancel/urge/supplement/delegate)
   │
2. 平台预校验（RBAC）
   ├── 查询用户角色
   ├── 检查角色是否在流程白名单
   ├── 通过 → 继续
   └── 拒绝 → 返回拒绝原因
   │
3. 属性校验（ABAC）[P2]
   ├── 加载属性规则
   ├── 评估规则表达式
   ├── 通过 → 继续
   └── 拒绝 → 返回拒绝原因
   │
4. OA实时校验 [P1]
   ├── 调用OA权限接口
   ├── 通过 → 允许操作
   └── 拒绝 → 返回OA拒绝原因
   │
5. 记录决策日志
   ├── 决策结果（allow/deny）
   ├── 命中的策略ID
   ├── 证据字段
   └── 拒绝原因
```

### 决策输出

```json
{
  "allowed": false,
  "reason": "您当前角色（教师）没有大额采购申请权限",
  "platformCheck": {
    "passed": false,
    "policyId": "POLICY-PURCHASE-LARGE",
    "reason": "角色teacher不在流程PURCHASE_LARGE白名单",
    "evidence": {
      "userRoles": ["teacher"],
      "requiredRoles": ["dean", "super_admin"]
    }
  },
  "oaCheck": {
    "passed": true,
    "reason": "OA check skipped (platform denied)"
  },
  "suggestion": "联系部门主任代为发起，或申请权限升级"
}
```

## 8.6 操作权限约束

### 办事动作权限

| 动作 | 权限要求 | 状态约束 | 二次确认 |
|------|----------|----------|----------|
| 发起申请 | 角色在流程白名单 | 无 | 否 |
| 查看进度 | 本人或管理员 | 无 | 否 |
| 撤回 | 本人 | SUBMITTED/IN_PROGRESS | 是 |
| 催办 | 本人 | SUBMITTED/IN_PROGRESS | 是 |
| 补件 | 本人 | REJECTED | 否 |
| 转办 | 本人 + 转办权限 | SUBMITTED/IN_PROGRESS | 是 |

### 数据可见性

| 数据 | Applicant | FlowAdmin | TenantAdmin | Auditor |
|------|-----------|-----------|-------------|---------|
| 自己的申请 | ✅ | ✅ | ✅ | ✅ |
| 他人的申请 | ❌ | ✅(本部门) | ✅(全部) | ✅(全部) |
| 审计日志 | ❌ | ❌ | ✅ | ✅ |
| 系统配置 | ❌ | ❌ | ✅ | ❌ |

## 8.7 认证方案

### MVP阶段：Mock认证

```
请求头：
  X-User-Id: <user_id>
  X-Tenant-Id: <tenant_id>

中间件：
  从请求头提取用户信息
  查询数据库获取用户角色
  注入到请求上下文
```

### 生产环境：JWT认证

```
登录流程：
  POST /api/v1/auth/login
  请求：{ username, password }
  响应：{ accessToken, refreshToken, expiresIn }

Token刷新：
  POST /api/v1/auth/refresh
  请求：{ refreshToken }
  响应：{ accessToken, expiresIn }

JWT Payload：
  {
    "sub": "user_id",
    "tenant_id": "tenant_id",
    "username": "zhangsan",
    "roles": ["teacher"],
    "iat": 1234567890,
    "exp": 1234571490
  }

Token有效期：
  accessToken: 1小时
  refreshToken: 7天
```
